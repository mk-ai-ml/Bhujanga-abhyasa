📋 Types of Errors

Python distinguishes between two main categories of errors: syntax errors that prevent code from running at all, and runtime errors (exceptions) that occur while the program is executing. Understanding the exception hierarchy is crucial for robust error handling.

Syntax vs. Runtime Errors

Syntax errors are caught by Python before execution. Runtime errors (exceptions) occur during program execution.

error_types.py
# Syntax Error - Code won't run at all
# Missing colon would cause SyntaxError
# if x > 5  # <-- SyntaxError!

# Runtime Errors - Exceptions that occur during execution
x = 'hello'
result = x + 5  # TypeError: can't add string and int

items = [1, 2, 3]
value = items[10]  # IndexError: list index out of range

data = {'name': 'Alice'}
age = data['age']  # KeyError: 'age'

number = int('not a number')  # ValueError: invalid literal

Exception Hierarchy

Python organizes exceptions in a hierarchy. At the top is BaseException, with most errors inheriting from Exception.

BaseException
├── SystemExit
├── KeyboardInterrupt
└── Exception (all typical errors inherit from this)
    ├── TypeError
    ├── ValueError
    ├── KeyError
    ├── IndexError
    ├── RuntimeError
    ├── ZeroDivisionError
    └── ... (200+ standard exceptions)
Pro Tip

Always catch the most specific exception first. Catch ValueError before Exception. This lets you handle different errors appropriately.

🔄 try/except Basics

The try/except block is the foundation of error handling. The code in the try block is executed, and if an exception occurs, it's caught and handled by the except block.

try_except_basic.py
# Simple try/except
try:
    age = int(input('Enter your age: '))
    print(f'You are {age} years old')
except:
    print('Please enter a valid number')

# Multiple except clauses
try:
    result = 10 / int(input('Divisor: '))
    print(result)
except ValueError:
    print('Invalid input: not a number')
except ZeroDivisionError:
    print('Cannot divide by zero')

Avoid Bare except

Never use a bare except: clause without specifying an exception type. It catches everything, including system exits and keyboard interrupts, making debugging harder.

✅ The else Clause

The else clause in a try block executes only if no exception occurs. Use it for code that should only run when the try block succeeds.

try_else_example.py
# Using else with try/except
try:
    num = int(input('Enter a number: '))
except ValueError:
    print('That was not a valid number')
else:
    # This runs only if no exception occurred
    squared = num * num
    print(f'{num} squared is {squared}')

# Practical: Process file only if it opens successfully
try:
    file = open('data.txt', 'r')
except FileNotFoundError:
    print('File not found')
else:
    content = file.read()
    print(content)
When to Use else
  • Code that depends on the try block succeeding
  • Operations that should only happen if no error occurred
  • Keeps exception handling separate from success logic

🔒 finally

The finally block executes unconditionally, whether an exception occurred or not. It's essential for cleanup tasks like closing files, releasing resources, or completing transactions.

finally_example.py
# finally always runs
try:
    file = open('data.txt', 'r')
    data = file.read()
    print(data)
except FileNotFoundError:
    print('File not found')
finally:
    print('Closing file...')  # Runs regardless
    file.close()

# Database connection cleanup
try:
    db = connect_database()
    db.execute('INSERT INTO users VALUES (...)')
except DatabaseError:
    print('Database error occurred')
finally:
    db.close()  # Always close connection

Use finally for cleanup: closing files, releasing locks, committing/rolling back transactions. Even if an exception occurs mid-operation, the cleanup code will run.

📝 Exception as Variable

Capture exception details using the as keyword to access error messages, tracebacks, and other information about what went wrong.

exception_variable.py
# Capture exception as variable
try:
    age = int(input('Age: '))
except ValueError as e:
    print(f'Error: {e}')  # Print error message

# Access full error information
try:
    result = 10 / 0
except ZeroDivisionError as error:
    print(f'Exception type: {type(error)}')
    print(f'Error message: {error}')

# Log error with context
try:
    data = load_data('user.json')
except Exception as err:
    print(f'Failed to load data: {err}')
    print(f'Error type: {err.__class__.__name__}')
Exception Object Properties
  • str(e) - Error message string
  • type(e) - Exception class
  • e.args - Tuple of arguments passed to exception
  • e.__traceback__ - Stack trace information

⬆️ Raising Exceptions

Use the raise keyword to manually trigger exceptions when your code detects an invalid state. This signals to calling code that something went wrong.

raising_exceptions.py
# Raise exception with message
def validate_age(age):
    if age < 0:
        raise ValueError('Age cannot be negative')
    if age > 150:
        raise ValueError('Age seems unrealistic')
    return age

# Re-raise exception
try:
    user_age = int(input('Age: '))
    validate_age(user_age)
except ValueError as e:
    print(f'Validation error: {e}')
    raise  # Re-raise same exception

# Catch and re-raise different exception
try:
    result = risky_operation()
except IOError as e:
    print(f'IO failed: {e}')
    raise RuntimeError('Operation failed')
Be Specific When Raising

Always raise specific exception types with clear messages. This helps calling code understand what went wrong and handle it appropriately.

🏗️ Custom Exceptions

Create your own exception classes by inheriting from Exception. This lets you define domain-specific errors that reflect your application's logic.

custom_exceptions.py
# Define custom exception classes
class InsufficientFundsError(Exception):
    """Raised when account has insufficient balance"""
    pass

class InvalidAccountError(Exception):
    """Raised when account number is invalid"""
    def __init__(self, account_id):
        self.account_id = account_id
        super().__init__(f'Invalid account: {account_id}')

# Use custom exceptions
def withdraw(account_id, amount):
    if account_id not in accounts:
        raise InvalidAccountError(account_id)

    balance = accounts[account_id]
    if balance < amount:
        raise InsufficientFundsError('Not enough money')

    return balance - amount

# Catch custom exceptions
try:
    withdraw('ACC123', 1000)
except InvalidAccountError as e:
    print(f'Bad account: {e.account_id}')
except InsufficientFundsError:
    print('Cannot complete withdrawal')
Exception Hierarchy Pattern

Create a base exception for your application, then inherit specific exceptions from it. This lets callers catch all your app's errors with one clause: except AppError:

⛓️ Exception Chaining

When handling an exception and raising a new one, preserve the original error context using the from keyword or the implicit exception chain. This maintains the full error history for debugging.

exception_chaining.py
# Explicit chaining with 'from'
try:
    file = open('missing.txt')
except FileNotFoundError as original_error:
    raise RuntimeError('Cannot load config') from original_error

# Access the cause
try:
    process_data()
except ValueError as e:
    original = e.__cause__  # Exception that caused this
    context = e.__context__  # Implicit exception context

# Suppress exception context if not relevant
try:
    result = fetch_data()
except NetworkError:
    raise DataError('Data unavailable') from None
Exception Attributes
  • __cause__ - Exception explicitly raised with from
  • __context__ - Exception that implicitly occurred
  • __suppress_context__ - Whether to show context in traceback

📖 Context Managers

Context managers (the with statement) automatically handle setup and cleanup. They're perfect for resource management and ensure cleanup happens even if an exception occurs.

context_managers.py
# Using with statement for file handling
with open('data.txt', 'r') as file:
    content = file.read()
# File is automatically closed here

# Multiple resources
with open('input.txt') as src, \
     open('output.txt', 'w') as dst:
    for line in src:
        dst.write(line.upper())

# Create custom context manager
class DatabaseConnection:
    def __enter__(self):
        print('Connecting...')
        return self

    def __exit__(self, exc_type, exc_val, exc_tb):
        print('Disconnecting...')
        return False  # Don't suppress exceptions

with DatabaseConnection() as db:
    db.execute('SELECT * FROM users')
__enter__ and __exit__
  • __enter__ - Called when entering with block, returns resource
  • __exit__ - Called on exit (normal or exception), does cleanup
  • Return True from __exit__ to suppress exceptions

✔️ Validation Patterns

Use validation patterns to catch errors early and provide clear feedback. Guard clauses, assertions, and early returns keep code clean and errors predictable.

validation_patterns.py
# Guard clause pattern
def process_user(user_data):
    if user_data is None:
        raise ValueError('User data required')
    if 'name' not in user_data:
        raise KeyError('Name field missing')
    if len(user_data['name']) == 0:
        raise ValueError('Name cannot be empty')

    # Main logic starts here
    return save_to_database(user_data)

# Assert pattern for debugging
def divide(a, b):
    assert b != 0, 'Divisor cannot be zero'
    return a / b

# Type checking
def increment(value):
    if not isinstance(value, int):
        raise TypeError(f'Expected int, got {type(value)}')
    return value + 1
Fail Fast, Fail Clear

Validate inputs at function entry. Raise specific exceptions with clear messages. This makes bugs easier to find and prevents invalid data from corrupting your system.

🌍 Real-World Patterns

Practical patterns for production code: retry logic, graceful degradation, error accumulation, and logging. These techniques make systems robust and maintainable.

Retry Pattern with Exponential Backoff

Automatically retry failed operations with increasing delays.

retry_pattern.py
import time

def retry_with_backoff(func, max_attempts=3):
    attempt = 1
    delay = 1  # Start with 1 second

    while attempt <= max_attempts:
        try:
            return func()
        except Exception as e:
            if attempt == max_attempts:
                raise  # Give up on last attempt
            print(f'Attempt {attempt} failed, retrying in {delay}s')
            time.sleep(delay)
            delay *= 2  # Exponential backoff
            attempt += 1

Graceful Degradation

Provide fallback behavior when operations fail.

graceful_degradation.py
def get_user_data(user_id):
    try:
        # Try to fetch from primary database
        return primary_db.get(user_id)
    except DatabaseError:
        try:
            # Fall back to cache
            return cache.get(user_id)
        except CacheError:
            # Last resort: return default
            return {'id': user_id, 'status': 'unknown'}

# Error accumulation pattern
def validate_users(users):
    errors = []

    for user in users:
        try:
            validate_user_data(user)
        except ValidationError as e:
            errors.append({'user_id': user['id'], 'error': str(e)})

    if errors:
        raise ValidationError(f'{len(errors)} users failed validation')

    return True

Logging Exceptions

Log detailed error information for debugging and monitoring.

logging_exceptions.py
import logging

logger = logging.getLogger(__name__)

def process_payment(payment_id):
    try:
        result = charge_card(payment_id)
        logger.info(f'Payment {payment_id} processed')
        return result
    except CardDeclinedError as e:
        logger.warning(f'Card declined: {e}')
        return {'status': 'declined'}
    except Exception as e:
        logger.error(f'Payment failed: {e}', exc_info=True)
        raise
Production Readiness

In production code, always log exceptions with full context. Use structured logging (JSON format) for easier analysis. Implement retry logic for transient failures and graceful degradation for resilience.
